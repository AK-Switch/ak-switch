## 📝 待议

- **全 Key 熔断错误提示** — 某 provider 所有 Key 都熔断时，是否应返回类似 `"sensenova 所有 API Key 已熔断"` 的明确错误信息？（参考 ccswitch 的"所有供应商已熔断，无可用渠道"）
- 关于更改项目名以及整个 IP 的名字，会很麻烦吗？我想改名为 AK switch，也就是 API key switch 的简称。我们现在整个 IP 的名字是不是硬编码了呀？是不是不方便？一般来说，大家都是怎么处理这个品牌名字的？如果说频繁变动的话，是不是硬编码就不方便了？包括GitHub仓库的名字以及URL的地址,这些能不能方便地更改
- 关于这个日志系统，我的理解是现在只有运行时的日志，是不是？那如果崩溃之后，服务被关停了，我们是不是什么日志都查不到？你觉得有必要做一些持久化的日志吗？那又如何处理持久化日志和运行时日志之间的关系呢？
- 锦上添花的一点是，关于运行时的日志输出，可不可以添加一些颜色？比如说那些，你懂吧，我们可以来讨论。
- 我对它将每个请求均匀分配到每个 API key 的机制很好奇，想了解以下两点：
    1. 这是不是目前的最优解？有没有更好的解法？
    2. 加了我们这一层 API key 之间的轮转之后，会比直接直连增加多少延迟？
- 包括关于重试的机制，比如遇到 429 之后的重试机制。我不清楚现在是一个怎样的机制，但我倾向于碰到 429 之后，就不要在短时间内再去重试了，把时间拉长一点。而且，这些时间都是要弄成可以自定义的配置。
- 还有就是关于key达到了今日限额之后的长期禁用是会被自动解禁的吗? 就类似于隔一段时间检查一次健康, 这样就可以避免他恢复额度之后我们又没能及时使用
